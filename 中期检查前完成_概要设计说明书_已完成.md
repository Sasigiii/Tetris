### 1.2 背景

随着数字化教育的发展，传统机械记忆单词的方式在用户体验与学习效率方面逐渐暴露局限性。基于游戏化机制的学习系统可以在“挑战—反馈—强化”循环中提升用户参与度，进而改善词汇学习的持续性和有效性。  

本项目以 Unity 为客户端开发平台，围绕“单词填空 + 俄罗斯方块式交互”设计玩法：玩家通过控制十字方块下落，将正确字母填入空缺位置，完成一组单词后进入下一组，最终形成关卡化学习闭环。  

系统总体目标如下：

- 实现词库驱动的关卡生成机制；
- 实现可操作、可反馈、可结算的核心玩法流程；
- 实现学习辅助机制（如开局单词提示）；
- 实现音频反馈与参数设置机制；
- 实现线性进度管理与可持续扩展架构。

项目基本信息说明如下：

a) 软件名称：Tetris（英语单词学习游戏）  
b) 项目提出者/开发者：毕业设计项目组（学生本人）  
c) 软件应用范围与用户：面向中学及大学英语学习者，支持词库选择、关卡挑战和学习反馈。

### 1.3 术语

为避免歧义，现对本文档涉及的关键术语进行统一定义：

- MVC（Model-View-Controller）：一种经典界面分层模式，用于实现数据、视图与业务控制的解耦。  
- UIManager：项目中的界面调度中心，采用栈式结构管理面板生命周期。  
- 普通面板（Normal Panel）：Push 时会暂停下层面板，Pop 时恢复下层面板。  
- 弹出面板（Popup Panel）：叠加显示于下层界面，Push/Pop 不改变下层显示状态。  
- LRUObjectPool：基于“最近最少使用”策略的对象池组件，控制内存与实例数量。  
- IPoolable：对象池生命周期接口，定义 `OnPoolGet` 与 `OnPoolRelease` 两个标准回调。  
- WordHintUI：关卡正式开始前的单词提示面板，提供倒计时预览。  
- GamePlayUI：核心交互界面，承载方块控制、单词填充、分数变化等逻辑。  
- GameOverUI：结算界面，展示分数、星级与结果反馈。  
- LexiconDatabase：基于 SQLite 的词库访问层。  
- AudioManager：统一音频服务，负责 BGM、事件音效播放与音量管理。

## 2. 总体设计

### 2.1 系统体系结构

系统采用“数据层—业务层—表现层—公共服务层”四层结构，以实现职责清晰、边界明确、便于演进的工程目标。各层关系如下一致：上层可依赖下层，下层不反向依赖上层，公共服务层向多层提供横切能力。  

图2-1 系统体系结构图如下所示：

```mermaid
flowchart TB
    U[玩家输入/操作]
    UI[表现层<br/>MainUI / LexiconUI / ChooseUI / GamePlayUI / WordHintUI / GameOverUI / SettingsUI]
    BIZ[业务层<br/>GamePlayManager / ScoreManager / ProgressManager / LexiconConfig]
    DATA[数据层<br/>LexiconDatabase(SQLite) / PlayerPrefs]
    INFRA[公共服务层<br/>UIManager / AudioManager / LRUObjectPool / GameContext]

    U --> UI
    UI --> BIZ
    BIZ --> DATA
    UI --> INFRA
    BIZ --> INFRA
    INFRA --> DATA
```

从该图可以看出，表现层负责用户交互与界面呈现，业务层负责规则计算与流程控制，数据层负责持久化访问，公共服务层贯穿各层提供导航、音频和对象复用等基础能力。此结构在保证实现效率的同时，也为后续模块替换与功能扩展提供了稳定边界。

#### （1）数据层

数据层承担词汇数据存取与配置持久化职责，包含：

- **SQLite 词库**：按词库类型分表（小学/初中/高中/CET4/CET6），核心字段包括 `wordRank`、`headWord`、`tranCn`。  
- **配置与进度存储**：使用 `PlayerPrefs` 保存通关进度、BGM 开关及各事件音量等参数。  

数据层设计原则为“查询接口稳定、存储细节内聚”，对上层统一暴露业务友好的方法（如按关卡分页取词），避免上层直接拼接 SQL。

#### （2）业务层

业务层负责游戏规则与状态控制，主要组件如下：

- `GamePlayManager`：核心状态机控制器，驱动生成、下落、碰撞、校验、通关/失败流转；
- `ScoreManager`：分数增减规则与失败阈值判定；
- `ProgressManager`：线性解锁逻辑与持久化更新；
- `LexiconConfig`：词库参数配置（最大词长、每关单词数）。

业务层将“规则”从“表现”中抽离，保证玩法变化不会大面积影响 UI 代码。

#### （3）表现层

表现层负责玩家交互反馈和信息呈现，基于 UGUI + MVC 架构实现：

- 主流程界面：MainUI、LexiconUI、ChooseUI、GamePlayUI、GameOverUI；  
- 辅助界面：SettingsUI（音频设置）、WordHintUI（关卡预览）。  

在表现层中，Controller 仅组织视图行为与业务调用，不承载底层规则；View 聚焦组件引用；Model 聚焦页面状态。

#### （4）公共服务层

公共服务层作为基础设施，向全局提供横向能力：

- `UIManager`：统一界面导航与生命周期管理；
- `AudioManager`：统一音频播放与音量参数管理；
- `LRUObjectPool` + `IPoolable`：统一对象复用规范；
- `GameContext`：运行态共享上下文（当前词库、当前关卡、数据库入口）。

综合而言，本系统总体架构既满足课程设计项目规模下的开发效率需求，又具备后续扩展为更完整产品形态的结构基础。

### 2.2 系统功能结构

本系统围绕“学习目标驱动的关卡循环”展开，功能上可拆分为启动、准备、挑战、反馈、设置五个阶段：

图2-2 系统功能结构图如下所示：

```mermaid
flowchart LR
    A[启动与导航<br/>MainUI→LexiconUI→ChooseUI]
    B[数据准备<br/>读取词库配置/生成关卡词汇]
    C[关卡预备<br/>显示WordHintUI+10秒倒计时]
    D[核心挑战<br/>方块移动/旋转/碰撞/填充]
    E[结算反馈<br/>GameOverUI分数与星级]
    F[设置管理<br/>SettingsUI(BGM与事件音量)]

    A --> B --> C --> D --> E
    A -. 可随时进入 .-> F
    E -. 返回选关继续挑战 .-> A
```

图2-2体现了系统“学习—挑战—反馈—再学习”的闭环流程：用户先完成词库与关卡选择，再通过预览阶段建立短时记忆，随后进入挑战阶段进行交互式填空，最终通过结算反馈强化学习结果。设置管理作为横向功能模块可在主流程中随时介入，提升个性化使用体验。

1. **启动与导航阶段**  
   游戏启动后加载 MainUI，用户选择进入词库界面，并在选关界面进行关卡选择。  

2. **数据准备阶段**  
   根据当前词库与关卡配置计算本关词汇集合，生成用于预览与玩法的基础数据。  

3. **预备学习阶段**  
   进入 GamePlayUI 后先显示 WordHintUI，提供倒计时（可提前跳过），帮助用户在压力较低状态下进行记忆。  

4. **核心挑战阶段**  
   借助十字方块下落机制执行字母填充，系统根据匹配结果进行得分或扣分，并在满足条件时判定通关或失败。  

5. **结果反馈阶段**  
   弹出 GameOverUI，展示分数与星级；通关时播放特效与音效并记录进度，失败时仅展示失败结果。  

此外，系统提供 **设置管理功能**，用户可在 SettingsUI 中配置 BGM 开关与事件音量，配置即时生效且可持久化。

### 2.4 系统关键技术

#### 2.4.1 MVC 与栈式 UI 管理

通过 MVC 对 UI 职责进行拆分，避免业务逻辑与组件引用耦合；通过栈式调度保证页面切换语义清晰，便于实现“返回上一层”等一致行为。

#### 2.4.2 Popup 面板机制

系统新增了 UI 类型判定（Normal/Popup），使 SettingsUI、WordHintUI 可悬浮显示而不破坏下层界面状态。该机制是本项目在交互体验上的关键增强点。

#### 2.4.3 对象池统一协议

采用 LRUObjectPool 管理复用对象，并以 `IPoolable` 标准化生命周期回调，兼顾性能与状态一致性，减少频繁实例化带来的卡顿和 GC 开销。

#### 2.4.4 词库查询与关卡数据组织

基于 SQLite 进行分页查询（`LIMIT + OFFSET`），按词库配置控制每关词量与词长，保证关卡难度可配置且生成过程可复现。

#### 2.4.5 DOTween 动画反馈

通过 DOTween Sequence 实现星级依次弹出、按钮压缩反馈、分数飘字等动态效果，提高可视化反馈质量与交互确定性。

#### 2.4.6 音频事件映射与独立音量控制

AudioManager 支持按事件名管理音量（`vol_{eventName}`），并与设置界面双向联动，保证玩家可针对具体音效类型进行精细调节。

### 3.2 模块：UI 框架与导航模块（DS_UI_01）

#### 3.2.1 模块编号和功能描述

编号：DS_UI_01。  
该模块负责所有界面资源的统一注册、按需加载、实例缓存、生命周期调度和导航控制。其核心目标是将“业务流程跳转”与“具体界面实例管理”解耦，降低页面之间的直接依赖关系。  

模块支持两类界面行为：

- 普通面板：覆盖式切换，强调流程推进；
- Popup 面板：叠加式展示，强调辅助信息或配置操作。

#### 3.2.2 操作者

- 直接操作者：各 UI Controller（发起 Push/Pop 调用）；  
- 间接操作者：玩家（通过按钮触发控制器行为）；  
- 系统操作者：启动器（首界面初始化）。

#### 3.2.3 与本模块相关的码表和表

- `UIConfig.json`：保存面板名称与 Prefab 路径映射关系。  
- 运行时缓存表：`_panelCache`（面板实例缓存字典）、`_panelStack`（导航栈）。

#### 3.2.4 界面设计与说明

UI 采用“控制器驱动视图”的设计方式，View 不直接调用其他页面。  
对于 Popup 页面，设计要求为：

1. Push 时不隐藏下层；
2. Pop 时不触发下层 Resume；
3. 业务侧可通过回调完成结果回传。

#### 3.2.5 输入信息

- 页面标识字符串（如 `GamePlayUI`）；  
- 控制器/视图/模型泛型类型；  
- 生命周期事件（OnEnter/OnPause/OnResume/OnExit）。

#### 3.2.6 输出信息

- 页面显示状态变化；  
- 栈状态变化；  
- 生命周期回调触发结果。

#### 3.2.7 算法

核心算法为“栈式调度 + 字典缓存”：

- 栈结构保证后进先出，天然匹配页面返回语义；
- 字典缓存减少重复实例化开销；
- 转场锁 `_isTransitioning` 防止短时间重复点击导致并发切换异常。

#### 3.2.8 处理流程

1. 接收 Push 请求，校验配置并加载或复用目标页面；  
2. 依据目标页面类型决定是否暂停当前栈顶；  
3. 将目标控制器入栈并触发 `OnEnter`；  
4. 接收 Pop 请求，弹出栈顶并触发 `OnExit`；  
5. 若弹出页面为普通类型，则恢复新栈顶 `OnResume`。  

#### 3.2.9 类设计

- `UIManager`：导航主控类；  
- `BaseController`：抽象控制器基类，包含 `IsPopup` 属性；  
- `BaseController<TView,TModel>`：泛型基类，统一初始化流程；  
- `BaseView`：统一页面显隐行为。

### 3.3 模块：词库与关卡数据模块（DS_DATA_02）

#### 3.3.1 模块编号和功能描述

编号：DS_DATA_02。  
该模块负责词库数据读取、条件筛选和关卡输入数据组织。通过统一的数据访问接口，确保玩法模块与数据库实现细节隔离。  

#### 3.3.2 操作者

- `ChooseUIController`：用于计算可选关卡数；  
- `WordHintUIController`：用于预览列表加载；  
- `GamePlayManager`：用于关卡词条初始化。

#### 3.3.3 与本模块相关的码表和表

- 词库表：`ChuZhong`、`GaoZhong`、`CET4`、`CET6` 等；  
- 核心字段：  
  - `wordRank`：词条序号；  
  - `headWord`：英文单词；  
  - `tranCn`：中文释义。  

#### 3.3.4 输入信息

- 词库类型（枚举）；  
- 当前关卡编号（int）；  
- 每关单词数量（int）；  
- 词长约束参数（来源于配置）。

#### 3.3.5 输出信息

- 本关单词集合（`List<WordEntry>`）；  
- 词库统计信息（词条总数、可用词数）。

#### 3.3.6 算法

模块采用分页查询策略：

`offset = (level - 1) * wordsPerLevel`  
`SELECT ... LIMIT wordsPerLevel OFFSET offset`

该方法计算复杂度稳定、实现简单，适合本项目规模并可保持关卡内容稳定。

### 3.4 模块：核心玩法模块（DS_GAME_03）

#### 3.4.1 模块编号和功能描述

编号：DS_GAME_03。  
该模块负责十字方块交互、网格填充判定、分数管理与关卡状态流转，是系统的核心业务引擎。  

#### 3.4.2 操作者

玩家（键盘输入 A/D/方向键/空格等）与 `GamePlayManager` 自动调度逻辑。

#### 3.4.3 输入信息

- 用户输入事件（移动、旋转、快速下落）；  
- 当前目标行和空位信息；  
- 当前方块底部字母与位置；  
- 分数阈值与关卡进度状态。

#### 3.4.4 输出信息

- 网格显示变化（填充成功/失败）；  
- 分数变化事件（加分/扣分）；  
- 游戏结束事件（通关或失败）；  
- 结果界面入参（最终分数、星级）。

#### 3.4.5 算法

系统采用“状态机 + 离散网格坐标映射”方式实现玩法逻辑：

- 状态机确保流程有序（加载、生成、下落、检测、结算）；  
- 碰撞检测基于目标行与当前底部格子的映射关系；  
- 匹配条件严格定义为“空位且字母一致”；  
- 错误填充触发扣分，低于阈值则进入失败结算。

#### 3.4.6 类设计

- `GamePlayManager`：玩法总控；  
- `CrossBlock`：方块数据与动作实体；  
- `WordGrid`：网格显示与数据映射；  
- `ScoreManager`：分数规则与星级计算。  

### 3.5 模块：关卡预览模块（DS_HINT_04）

#### 3.5.1 模块编号和功能描述

编号：DS_HINT_04。  
该模块在正式玩法开始前向玩家展示本关词汇，提供短时记忆窗口，降低上手门槛，提高学习属性与玩法衔接度。  

#### 3.5.2 操作者

玩家、`WordHintUIController`、`GamePlayUIController`。

#### 3.5.3 输入信息

- 当前关卡单词列表（英文 + 中文）；  
- 倒计时参数（默认 10 秒）；  
- 用户中断输入（ESC）。

#### 3.5.4 输出信息

- 预览列表渲染结果；  
- 倒计时文本输出（格式：`倒计时：N`）；  
- 完成回调 `OnHintFinished`（触发玩法启动）。

#### 3.5.5 处理流程

1. GamePlayUI 入场后先清空玩法网格显示；  
2. Push WordHintUI 并加载列表；  
3. 计时结束或用户跳过后 Pop WordHintUI；  
4. 触发回调初始化并启动 `GamePlayManager.StartLevel()`。

### 3.6 模块：音频与设置模块（DS_AUDIO_05）

#### 3.6.1 模块编号和功能描述

编号：DS_AUDIO_05。  
模块负责统一音频播放策略、事件映射及音量参数管理，目标是在性能可控前提下提供清晰的听觉反馈与用户可调参数体验。  

#### 3.6.2 操作者

- 玩家：通过 SettingsUI 调整音量与开关；  
- 业务模块：通过 `PlayEvent` 触发对应音效；  
- 控件模块：UIButton 点击音效调用。

#### 3.6.3 输入信息

- 音频事件名（`blockMove`、`fillCorrect`、`fillWrong`、`starPop`、`gameOverWin`、`gameOverLose`、`uiClick`）；  
- 事件音量（0~1）；  
- BGM 开关状态（bool）；  
- 资源路径（`Resources` 下音频名）。

#### 3.6.4 输出信息

- 音频播放行为；  
- 参数持久化结果；  
- 对象池复用结果（获取/回收）。

#### 3.6.5 类设计

- `AudioManager`：统一服务入口；  
- `AudioConfig`：事件路径映射配置；  
- `AudioPoolItem`：池化音源包装类；  
- `SettingsUIController`：参数绑定与交互控制。

### 4.2 关键接口说明

#### 4.2.1 接口：`UIManager.PushPanel<TController,TView,TModel>(panelName)`

- 接口编号：IF_UI_01  
- 接口说明：根据配置加载/复用页面并执行入栈流程；  
- 输入参数：页面名字符串、泛型类型参数；  
- 输出：无（通过界面状态变化体现）；  
- 调用者：各 UIController；  
- 约束：`panelName` 必须在 `UIConfig.json` 注册；泛型类型需与页面组件一致；  
- 处理规则：若目标页面为 Popup，则不暂停下层页面。

#### 4.2.2 接口：`LexiconDatabase.GetPlayableWordsForLevel(lexicon, level, wordsPerLevel)`

- 接口编号：IF_DB_01  
- 接口说明：按词库与关卡号分页提取本关可玩词条；  
- 输入：词库枚举、关卡编号、每关词数；  
- 输出：`List<WordEntry>`；  
- 调用者：`WordHintUIController`、`GamePlayManager`；  
- 约束：数据库必须已初始化；`level >= 1`；`wordsPerLevel > 0`。  

#### 4.2.3 接口：`WordHintUIController.OnHintFinished`

- 接口编号：IF_HINT_01  
- 接口说明：WordHint 结束后的单次回调入口；  
- 输入：无；  
- 输出：无；  
- 调用者：`GamePlayUIController` 订阅；  
- 约束：回调触发后需主动置空，防止重复触发。  

#### 4.2.4 接口：`AudioManager.SetEventVolume(eventName, volume)`

- 接口编号：IF_AUDIO_02  
- 接口说明：设置某一音频事件的独立音量并持久化；  
- 输入：事件名、音量值（0~1）；  
- 输出：无；  
- 调用者：`SettingsUIController`；  
- 约束：音量值超界时需截断到合法范围。  

#### 4.2.5 接口：`LRUObjectPool.Get()` / `Release(obj)`

- 接口编号：IF_POOL_01  
- 接口说明：从池中获取对象或回收对象，自动触发 `IPoolable` 生命周期；  
- 输入：回收时传入对象引用；  
- 输出：获取时返回可用对象；  
- 调用者：列表渲染、音频播放、动态文本效果等模块；  
- 约束：仅对已由池创建的对象进行回收；超过容量按 LRU 淘汰。  

### 4.3 接口异常与容错设计

为提升系统鲁棒性，接口层加入以下防护策略：

1. **空引用防护**：服务入口均进行空实例校验并输出日志；  
2. **参数合法化**：音量参数采用 `Mathf.Clamp01`；  
3. **重复调用保护**：UI 转场采用 `_isTransitioning` 保护；  
4. **单次回调清理**：`OnHintFinished` 使用后置空；  
5. **资源缺失降级**：音频/页面资源缺失时记录警告并避免崩溃。

## 5. 设计结论

本项目的概要设计以“学习闭环”和“工程可实现性”作为双重目标，围绕数据、业务、界面、音频与基础设施完成了系统性设计。通过本阶段产出可得出以下结论：

1. **设计完整性较高**  
   系统已覆盖从“词库读取—关卡准备—核心玩法—结果反馈—配置管理”的全流程设计，功能链路闭环明确。  

2. **架构可维护性良好**  
   采用 MVC + UI 栈式调度 + 公共服务分层，模块边界清晰，核心能力可复用，后续新增功能（新词库、新音效事件、新页面类型）具备较低改造成本。  

3. **性能与体验兼顾**  
   通过对象池、事件驱动与分层解耦，降低了运行时开销；通过预览面板、动画反馈和音频机制提升了用户学习体验。  

4. **扩展方向明确**  
   后续可在不破坏现有主结构的前提下，进一步扩展排行榜、学习统计、难度自适应、错题回顾等高级功能。  

综上，本概要设计具备现实可行性、工程可实施性与阶段评审所需的文档完整性，可直接作为下一阶段详细设计与编码实现的依据。
